# 预处理器测试 Web UI 规格说明

## 为什么
为了给 `preprocessor` 模块提供一个用户友好的 Web 测试界面。这将使开发和测试人员能更轻松地配置和运行各种测试场景，而无需依赖复杂的命令行参数，从而提高效率并减少错误。

## 将要做出的改动
- 将在一个全新的 `preprocessor_test_ui/` 目录下，创建一个新的、自包含的 Web 应用。
- 该应用将由一个简单的 FastAPI 后端和一个基于 React 的前端组成。
- **核心原则**: 后端服务**必须**仅作为 `preprocessor/main.py` 的命令行参数的“翻译层”，**严禁**在后端重新实现任何 `preprocessor` 已有的逻辑。
- **重大变更**: 无。这是一个纯新增的功能，不会对现有的 `preprocessor/main.py` 或 `run_pipeline.py` 的命令行功能产生任何影响。

## 影响
- **受影响的规格**: 无。这是一个新功能。
- **受影响的代码**: 所有新代码将被包含在 `preprocessor_test_ui/` 目录中。它将以只读方式与数据库 `exam_analysis.db` 交互，并作为子进程调用 `preprocessor/main.py`。

## 新增需求
### 需求: 预处理器测试 Web 界面
系统应提供一个基于 Web 的用户界面，用于配置和运行 `preprocessor` 模块的测试。**该界面的设计应遵循简单、大方的原则，不能没有样式。**

#### 场景: 用户通过 Web UI 运行一个混合模式的测试
- **假定** 用户已打开 Web UI 并已确认 UI 原型设计。
- **当** 用户选择“模拟测试”模式。
- **那么** UI 应显示标签：“如果选择模拟测试，请勾选需要 **模拟运行** 的步骤:”。
- **并且** 步骤 3、5、6 应被置灰，不可勾选。
- **并且** 步骤 1、2、4 应为可勾选状态。
- **并且当** 用户勾选了步骤 1 和 4 进行模拟运行。
- **并且当** 用户点击“开始测试”按钮。
- **那么** 后端服务**必须**将前端发送的配置，准确地翻译成 `preprocessor/main.py` 所支持的命令行参数。例如，它应构建一个类似 `python preprocessor/main.py --mock-case <默认测试案例> --real-steps 2` 的命令。
- **并且** 后端服务**必须**通过执行这个构建好的命令来启动测试，而不是在自身代码中实现测试逻辑。
- **并且** UI 应显示 `preprocessor/main.py` 进程的实时日志输出。
